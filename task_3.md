## 1. Отличия реляционных и нереляционных БД

### Реляционные БД (SQL)
- **Структура**: Данные организованы в таблицы (отношения) с строгой схемой, связанные ключами
- **Примеры**: MySQL, PostgreSQL, Oracle
- **Плюсы**:
  - Целостность данных (ACID-транзакции)
  - Сложные запросы (JOIN, агрегации)
  - Стандартизированный язык (SQL)
- **Минусы**:
  - Масштабирование (вертикальное)
  - Схема менее гибкая

### Нереляционные БД (NoSQL)
- **Структура**: Документы, ключ-значение, графы, колоночные
- **Примеры**: MongoDB, Cassandra, Redis
- **Плюсы**:
  - Горизонтальное масштабирование
  - Гибкая схема данных
  - Высокая производительность для конкретных сценариев
- **Минусы**:
  - Ограниченная поддержка транзакций
  - Менее стандартизированные запросы

### Отличия MySQL и PostgreSQL
- **MySQL**: Быстрее для чтения, проще в настройке, популярен для веб-приложений
- **PostgreSQL**: Более строгий стандарт SQL, расширенные типы данных, лучше для сложных операций

## 3. Гипотеза о типе БД

**Гипотеза**: Telegram использует гибридный подход с преобладанием нереляционных БД.

**Аргументация**:
1. **Масштаб**: Миллиарды сообщений ежедневно требуют горизонтального масштабирования (сильная сторона NoSQL)
2. **Структура сообщений**: Разнообразный контент (текст, фото, видео, файлы) лучше хранить в документ-ориентированной БД
3. **Скорость доступа**: Кэширование сессий, онлайн-статусов эффективнее в key-value хранилищах (Redis)
4. **Социальный граф**: Связи между пользователями, чатами, каналами могут использовать графовые БД
5. **Частичная реляционность**: Данные пользователей, настройки, billing могут использовать SQL для целостности

## 4. Ключевые функции и подходящие типы БД

### Функция 1: Обмен сообщениями
- **Данные**: Текст, метаданные (время, отправитель, чат), статус доставки
- **Тип БД**: **Документ-ориентированная NoSQL** (MongoDB)
- **Причина**: Гибкая схема для разных типов контента, горизонтальное масштабирование для хранения истории

### Функция 2: Управление контактами и чатами
- **Данные**: Списки контактов, участники чатов, права доступа
- **Тип БД**: **Графовая NoSQL** (Neo4j) + **Реляционная** для ACL
- **Причина**: Графы эффективны для социальных связей, SQL — для строгих прав доступа

### Функция 3: Медиаконтент (фото, видео, файлы)
- **Данные**: Файлы, превью, метаинформация
- **Тип БД**: **Объектное хранилище** (S3-совместимое) + **Key-value** для метаданных
- **Причина**: Эффективное хранение бинарных данных, быстрый доступ к метаданным

### Функция 4: Поиск по чатам и сообщениям
- **Данные**: Индексы сообщений, хэштеги, метаданные для поиска
- **Тип БД**: **Поисковый движок** (Elasticsearch)
- **Причина**: Оптимизирован для полнотекстового поиска, быстрого индексирования

### Функция 5: Пользовательские настройки и профили
- **Данные**: Профили пользователей, настройки приватности, сессии
- **Тип БД**: **Реляционная** (PostgreSQL) + **Key-value** для сессий (Redis)
- **Причина**: Целостность данных пользователей, транзакционность для настроек, скорость для сессий

## 5. Тест-кейсы для проверки типов БД

### Кейс 1: Целостность данных пользователя
- **Цель**: Проверить, что при обновлении email в профиле изменение отражается во всех связанных данных
- **Шаги**:
  1. Изменить email в настройках профиля
  2. Проверить email в чатах (как отправитель)
  3. Проверить email в уведомлениях
- **Ожидаемый результат**: Email одинаков во всех системах

### Кейс 2: Производительность получения истории чата
- **Цель**: Проверить скорость загрузки истории большого чата (10000+ сообщений)
- **Шаги**:
  1. Открыть чат с большой историей
  2. Замерить время загрузки первых 100 сообщений
  3. Прокрутить до начала истории, замерить время
- **Ожидаемый результат**: Загрузка < 2 сек для любого объема

### Кейс 3: Поиск по разнородному контенту
- **Цель**: Проверить поиск по тексту, хэштегам и медиа
- **Шаги**:
  1. Выполнить поиск по ключевому слову
  2. Поиск по хэштегу
  3. Поиск по названию файла
- **Ожидаемый результат**: Релевантные результаты из всех типов контента

### Кейс 4: Синхронизация между устройствами
- **Цель**: Проверить консистентность данных на разных устройствах
- **Шаги**:
  1. Отправить сообщение с телефона
  2. Проверить доставку на десктопе
  3. Удалить сообщение на одном устройстве
  4. Проверить удаление на другом
- **Ожидаемый результат**: Мгновенная синхронизация (< 1 сек)

## 6. Проверка гипотезы в Интернете (частичное совпадение).

**Результаты поиска "Telegram tech stack database":**

1. **Основная БД**: PostgreSQL для основных данных (пользователи, чаты, настройки)
2. **Сообщения**: Собственное решение на основе распределенного хранилища с шардированием
3. **Кэширование**: Redis для сессий, онлайн-статусов
4. **Медиа**: Распределенное файловое хранилище с репликацией между датацентрами
5. **Поиск**: Elasticsearch для индексации сообщений

**Что совпало**:
- Гибридный подход подтвердился
- Использование Redis для кэширования
- Распределенное хранение медиа
- PostgreSQL для структурированных данных

**Отличия от гипотезы**:
- Для сообщений используется не стандартная NoSQL, а кастомное решение
- Графовые БД явно не упоминаются в открытых источниках
- Акцент на максимальную безопасность и контроль над данными

Гипотеза была близка к реальности, но недооценила степень кастомных решений. Telegram, как security-focused приложение, разрабатывает многие компоненты самостоятельно для полного контроля над данными и их безопасностью.

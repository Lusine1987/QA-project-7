## 1. Изучение REST API

### 1.1 Методы REST API и их различия

**HTTP-методы в REST API** - это стандартизированные операции, которые определяют тип действия, выполняемого с ресурсом. Каждый метод имеет конкретную семантику и должен использоваться в соответствии с принципами REST.

#### Основные методы:

**1. GET - получение данных**
- **Назначение**: Чтение ресурсов
- **Идемпотентность**: Да (повторные идентичные запросы дают одинаковый результат)
- **Безопасность**: Да (не изменяет состояние сервера)
- **Тело запроса**: Не должно содержаться (но некоторые реализации допускают)
- **Тело ответа**: Содержит запрошенные данные
- **Пример**: Получение списка пользователей
```http
GET /api/users HTTP/1.1
```

**2. POST - создание данных**
- **Назначение**: Создание нового ресурса
- **Идемпотентность**: Нет (повторный запрос создаст еще один ресурс)
- **Безопасность**: Нет
- **Тело запроса**: Содержит данные для создания
- **Тело ответа**: Содержит созданный ресурс или ссылку на него
- **Пример**: Создание нового пользователя
```http
POST /api/users HTTP/1.1
Content-Type: application/json

{
  "name": "Иван Иванов",
  "email": "ivan@example.com"
}
```

**3. PUT - полное обновление**
- **Назначение**: Полная замена ресурса
- **Идемпотентность**: Да (повторные запросы дают тот же результат)
- **Безопасность**: Нет
- **Тело запроса**: Содержит полные новые данные ресурса
- **Тело ответа**: Может содержать обновленный ресурс
- **Пример**: Полное обновление данных пользователя
```http
PUT /api/users/123 HTTP/1.1
Content-Type: application/json

{
  "name": "Петр Петров",
  "email": "peter@example.com"
}
```

**4. PATCH - частичное обновление**
- **Назначение**: Частичное обновление ресурса
- **Идемпотентность**: Да (но зависит от реализации)
- **Безопасность**: Нет
- **Тело запроса**: Содержит только изменяемые поля
- **Тело ответа**: Может содержать обновленный ресурс
- **Пример**: Частичное обновление данных пользователя
```http
PATCH /api/users/123 HTTP/1.1
Content-Type: application/json

{
  "email": "newemail@example.com"
}
```

**5. DELETE - удаление данных**
- **Назначение**: Удаление ресурса
- **Идемпотентность**: Да (после первого удаления ресурс отсутствует)
- **Безопасность**: Нет
- **Тело запроса**: Обычно не содержит (может содержать в сложных случаях)
- **Тело ответа**: Обычно пустое или содержит статус
- **Пример**: Удаление пользователя
```http
DELETE /api/users/123 HTTP/1.1
```

**6. HEAD - получение заголовков**
- **Назначение**: Получение только заголовков (без тела)
- **Идемпотентность**: Да
- **Безопасность**: Да
- **Использование**: Проверка существования ресурса, получение метаданных

**7. OPTIONS - получение поддерживаемых методов**
- **Назначение**: Определение доступных операций для ресурса
- **Идемпотентность**: Да
- **Безопасность**: Да
- **Пример**: Проверка доступных методов для ресурса
```http
OPTIONS /api/users HTTP/1.1
```

### 1.2 HTTP-статус коды в тестировании.

HTTP-статус коды делятся на 5 классов, где первая цифра определяет категорию ответа:

#### 1xx - Информационные (Informational). Коды информируют о процессе обработки запроса.
#### 2xx - Успех (Success). Запрос был успешно получен, понят и обработан.
#### 3xx - Перенаправление (Redirection).Требуются дополнительные действия для завершения запроса.
#### 4xx - Ошибка клиента (Client Error). Запрос содержит неправильный синтаксис или не может быть выполнен.
#### 5xx - Ошибка сервера (Server Error). Сервер не смог выполнить допустимый запрос.
#### Практическое применение статус-кодов в тестировании:
1. **Позитивные тесты**: Проверка 2xx кодов при корректных запросах
2. **Негативные тесты**: Проверка 4xx кодов при некорректных запросах
3. **Тестирование граничных значений**: 413 при превышении размера, 429 при лимитах
4. **Тестирование безопасности**: 401 при отсутствии аутентификации, 403 при недостатке прав
5. **Тестирование доступности**: 503 при недоступности сервиса, 504 при таймаутах

### 1.3 JSON (JavaScript Object Notation)

**JSON** - это текстовый формат обмена данными, основанный на синтаксисе JavaScript, но независимый от языка программирования.

#### Структура JSON:
- **Объекты**: Набор пар ключ-значение в фигурных скобках `{}`
- **Массивы**: Упорядоченные списки значений в квадратных скобках `[]`
- **Значения**: Строки, числа, объекты, массивы, true, false, null

#### Типы данных в JSON:
1. **Строки** (string) - в двойных кавычках: `"текст"`
2. **Числа** (number) - целые или с плавающей точкой: `42`, `3.14`
3. **Объекты** (object) - коллекция пар ключ-значение: `{"ключ": "значение"}`
4. **Массивы** (array) - упорядоченная коллекция: `[1, 2, 3]`
5. **Булевы значения** (boolean): `true`, `false`
6. **null** - специальное значение: `null`

### 1.4 Преимущества JSON над XML и YAML

#### JSON vs XML:
| Параметр | JSON | XML |
|----------|------|-----|
| **Читаемость** | Высокая (близок к структурам данных языков программирования) | Низкая (много служебных символов) |
| **Объем данных** | Меньше (нет закрывающих тегов) | Больше (закрывающие теги увеличивают объем) |
| **Скорость обработки** | Выше (проще парсинг) | Ниже (сложный парсинг DOM/SAX) |
| **Поддержка типов данных** | Базовые типы (строки, числа, булевы) | Только строки (нужно преобразование) |
| **Поддержка комментариев** | Нет | Да (<!-- комментарий -->) |
| **Пространства имен** | Нет | Да (сложная система namespaces) |
| **Схемы валидации** | JSON Schema | XML Schema (XSD) |
| **JavaScript интеграция** | Нативная (JSON.parse(), JSON.stringify()) | Требует парсинга |

**Ключевые преимущества JSON:**
1. **Компактность**: Меньший объем данных за счет отсутствия закрывающих тегов
2. **Простота парсинга**: Нативная поддержка в JavaScript и многих других языках
3. **Читаемость**: Структура понятна как человеку, так и машине
4. **Быстрота передачи**: Меньший размер = быстрее передача по сети
5. **Универсальность**: Поддерживается всеми современными языками программирования

#### JSON vs YAML:
| Параметр | JSON | YAML |
|----------|------|------|
| **Синтаксис** | Строгий (фигурные и квадратные скобки) | Свободный (отступы и двоеточия) |
| **Комментарии** | Не поддерживает | Поддерживает (# комментарий) |
| **Человекочитаемость** | Хорошая | Отличная (близок к естественному языку) |
| **Сложность парсинга** | Проще (четкий синтаксис) | Сложнее (зависит от отступов) |
| **Использование** | API, передача данных | Конфигурационные файлы |
| **Безопасность** | Выше (строгий синтаксис) | Ниже (может выполнять код) |

**Почему JSON предпочтительнее для API:**
1. **Предсказуемость**: Строгий синтаксис уменьшает ошибки
2. **Безопасность**: Невозможность встраивания кода
3. **Стандартизация**: Единый формат для всех языков
4. **Производительность**: Быстрее обработка и передача

### 1.5 JSON Schema

**JSON Schema** - это язык описания структуры JSON-документов для валидации, документации и взаимодействия.

#### Основные возможности JSON Schema:
1. **Валидация структуры**: Проверка обязательных полей, типов данных
2. **Описание данных**: Документирование формата данных
3. **Гипермедиа**: Описание взаимодействий (HATEOAS)
4. **Автогенерация**: Генерация кода и форм

#### Пример JSON Schema:
```json
{
  "$schema": "http://json-schema.org/draft-07/schema#",
  "title": "Пользователь",
  "description": "Схема данных пользователя",
  "type": "object",
  "properties": {
    "id": {
      "type": "integer",
      "description": "Уникальный идентификатор"
    },
    "name": {
      "type": "string",
      "minLength": 2,
      "maxLength": 100
    },
    "email": {
      "type": "string",
      "format": "email"
    },
    "age": {
      "type": "integer",
      "minimum": 18,
      "maximum": 120
    }
  },
  "required": ["name", "email"],
  "additionalProperties": false
}
```

### 1.6 Сериализация и десериализация в контексте JSON

**Сериализация (маршалинг)** - процесс преобразования объекта в формат JSON для передачи или хранения.

**Десериализация (демаршалинг)** - обратный процесс преобразования JSON обратно в объект.

#### Процесс сериализации:
```javascript
// Исходный объект в памяти
const user = {
  id: 1,
  name: "Иван Иванов",
  active: true,
  roles: ["admin", "user"]
};

// Сериализация в JSON-строку
const jsonString = JSON.stringify(user);
// Результат: '{"id":1,"name":"Иван Иванов","active":true,"roles":["admin","user"]}'
```

#### Процесс десериализации:
```javascript
// JSON-строка (полученная из сети или файла)
const jsonString = '{"id":1,"name":"Иван Иванов","active":true,"roles":["admin","user"]}';

// Десериализация в объект
const userObject = JSON.parse(jsonString);
// Результат: объект со структурой как исходный
```

#### Особенности процессов:
1. **Потеря методов**: При сериализации теряются методы объекта
2. **Циклические ссылки**: Могут вызвать ошибки (нужна специальная обработка)
3. **Даты**: Преобразуются в строки (ISO формат)
4. **undefined**: Пропускается при сериализации
5. **Функции**: Не сериализуются

#### Оптимизации для API:
1. **Селективная сериализация**: Отправка только необходимых полей
2. **Кэширование**: Сохранение сериализованных данных
3. **Сжатие**: Использование gzip/brotli для JSON
4. **Стриминг**: Постепенная обработка больших JSON

## 4. Отличия в подходах к дизайну и структуре

### GitHub API:
- **Архитектура**: Комплексная система с полным CRUD
- **Аутентификация**: Обязательна для большинства операций
- **Версионирование**: Через заголовки или URL
- **Ограничения**: Строгие лимиты запросов в час
- **Документация**: Официальная документация с интерактивными примерами

### Dadata API:
- **Архитектура**: Специализированная, для обработки данных
- **Аутентификация**: Простой API-ключ
- **Версионирование**: Через префикс в URL (/api/4_1/)
- **Ограничения**: Лимиты по дням, более щадящие
- **Документация**: Документация с примерами на русском

### Особенности дизайна:

| Аспект | GitHub API | Dadata API |
|--------|------------|------------|

| **Стандарты** | Строгое следование REST | REST, но только POST для большинства endpoints |
| **Локализация** | Международная | Сфокусирована на российских данных |
| **Бизнес-модель** | Freemium (бесплатно для открытых репозиториев) | Freemium (10,000 бесплатных запросов/день) |

| `opf` | object | Объект с данными об организационно-правовой форме |
| `address` | object | Адрес организации |
| `state` | object | Статус организации |
| `management` | object | Информация о руководстве |

#### Вложенные объекты:

**Объект `name`:**
| Ключ | Тип | Описание |
|------|-----|----------|
| `full_with_opf` | string | Полное название с ОПФ |
| `short_with_opf` | string | Краткое название с ОПФ |
| `full` | string | Полное название |
| `short` | string | Краткое название |

**Объект `address`:**
| Ключ | Тип | Описание |
|------|-----|----------|
| `value` | string | Отформатированный адрес |
| `unrestricted_value` | string | Полный адрес |
| `data` | object | Детализированные данные адреса |

**Объект `state`:**
| Ключ | Тип | Описание |
|------|-----|----------|
| `status` | string | Статус организации (ACTIVE, LIQUIDATING, LIQUIDATED) |
| `registration_date` | number | Дата регистрации (timestamp) |
| `liquidation_date` | number | Дата ликвидации (timestamp) |

#### Особенности структуры данных:

1. **Иерархия**: Глубокая вложенность данных
2. **Нулевые значения**: Многие поля могут быть null
3. **Даты**: Представлены как timestamp в миллисекундах
4. **Локализация**: Все данные на русском языке
5. **Форматы**: Используются российские стандарты (ИНН, КПП, ОГРН)
